# Fresatanika (Carrd-based catalog → Node/Express)

- **better-sqlite3 does NOT compile in this Replit environment** (native build fails). Use a JSON-file DB instead. **Why:** repeated build failures wasted time; the JSON store also satisfied the client's "un archivo" (a single file) requirement.

- **All persistent state lives under `DATA_DIR`** (bookings JSON, uploaded photos, session secret). In dev this defaults to a local `data/` dir; in the Dockerfile it is `/data`.

- **CapRover deploy requires a persistent volume mounted at `/data`.** **Why:** without it, every redeploy wipes all bookings, uploaded photos, and the session secret. This is the single most important deploy step and is easy to forget.

- **Secrets are required, not defaulted.** `ADMIN_PASSWORD` (admin panel at `/admin`) and `SMTP_PASS` (Zoho `admin@fourone.com.do`) must be set as env/secrets. The admin password fallback was intentionally removed so a missing env fails safe instead of exposing a predictable password.

- **Wompi Web Checkout requires an integrity signature or it rejects the tx** with "Firma de integridad requerida no enviada". Append `&signature:integrity=<hash>` to the checkout URL, where `hash = SHA256(reference + amountInCents + currency + WOMPI_INTEGRITY_SECRET)` (hex). Compute it SERVER-SIDE in `/api/booking` (needs the secret) and return it in the JSON; the frontend only appends it. **Why:** the signed `amountInCents`/`reference`/`currency` MUST exactly equal what the redirect URL sends (both `precio*100` / `ref` / `'COP'`) or Wompi rejects it. The `test_integrity_*` secret pairs with `pub_test_*`, `prod_integrity_*` with `pub_prod_*` — swap BOTH together when going live. Only sign Wompi payments, not transferencia. Fail-closed: if `metodo=wompi` but the secret is unset, return 503 instead of redirecting to a checkout that will error.

- **Payment integrity:** `/api/booking` accepts a price only if it appears in the catalog (prices are parsed from `index.html` at startup). `/gracias/:ref` marks a booking as `agendado` ONLY when the Wompi tx is APPROVED **and** `reference`, `amount_in_cents` (== precio_cop*100), and `currency === 'COP'` all match exactly. Never loosen these checks.

- **Photo uploads are raster-only** (jpeg/png/webp; SVG blocked) and served with `X-Content-Type-Options: nosniff` + `Content-Security-Policy: default-src 'none'; sandbox` to prevent stored-XSS in the authenticated admin view.

- **CSRF on admin POST routes uses Origin/Referer validation** (a `mismoOrigen` middleware), not a token library. **Why:** the whole UI is plain HTML built via string concatenation in `lib/templates.js`, so per-form hidden-token plumbing is heavy; combined with the `SameSite=Lax` session cookie, Origin/Referer matching is sufficient defense for the destructive actions. **How to apply:** any new state-changing admin POST must be guarded by `mismoOrigen` (and `requiereAdmin`).

- **Any new uploaded-file field must also be threaded through the delete/cleanup path** in `lib/db.js` (`eliminarAdjuntos`, used by both `eliminarPorRef` and `eliminarPendientesAntiguos`), or the file becomes an orphan on disk. **Why:** `comprobante` was added everywhere except deletion, leaving sensitive payment-proof files behind after admin deletes a booking. **How to apply:** every field that stores an uploaded filename must be unlinked when its record is removed.

- **Scheduled reminders run in-process via `setInterval` in the `app.listen` callback** (like `limpiarPendientes`), guarded by an in-process boolean flag to avoid overlap on slow SMTP. Follow-up emails (5 semanas / 4 meses after `trabajo_hecho_en`) go to the ADMIN inbox (`NOTIFY_TO`/`SMTP_USER`), not the client — contacto is usually a social handle, not an email. **Why:** no external cron/queue in this CapRover setup. **How to apply:** persist a per-record "sent" timestamp (e.g. `seguimiento_5s_en`) set only after `r.enviado`, so a resend never fires; these internal tracking flags are the EXCEPTION to the six-place threading rule below — they are NOT rendered in any view/export.

- **Discounts (promotions/codes) apply to HECHIZOS ONLY, and the server must verify the booking really is that hechizo — not just trust the client-sent `hechizo_clave`.** The catalog is parsed from `#hechizos-section` of `index.html` at startup into a clave→hechizo map. Duplicate hechizo names with different prices exist (e.g. "Corte de Lazos"), so each map entry keeps a `precios` array of ALL its catalog prices; a discount is applied only if the client-sent base price is a member of that hechizo's `precios`. **Why:** without the price-membership check, a client could book a non-hechizo service at its own valid price but send a valid `hechizo_clave` and steal a hechizo-only discount. **How to apply:** never gate a discount on the clave alone; the (clave, base-price) pair must both match the catalog.

- **Discounts never stack — take the larger of date-promo % vs promo-code %** (`Math.max`). The final price is always recomputed server-side in `/api/booking`; the frontend only sends the ORIGINAL base price + clave + code and then uses the server's returned `precio_cop` for the Wompi `amount-in-cents`. Promo codes are admin-only to create/delete (`soloAdmin`); date promos and everything else are allowed for assistants.

- **Two admin roles are distinguished purely by which password matched at login** (`ADMIN_PASSWORD` → `admin`, `ASSISTANT_PASSWORD` → `asistente`), stored as `session.rol`; both compared timing-safe. Assistants are blocked from: creating/deleting promo codes, and deleting agendamientos already marked `trabajo_hecho` (enforced in BOTH the route and the UI). **How to apply:** gate admin-only actions with the `soloAdmin` middleware and mirror the restriction in `adminDashboard` so the UI matches the backend.

- **USD is a NATIVE catalog price, never a conversion. There is NO TRM/exchange-rate logic anywhere (fully removed).** Wompi charges COP only (confirmed official docs). Each service has independent COP and USD prices parsed from `index.html`. The charge depends on the currency+method combo, keyed in `server.js /api/booking` off `usdManual = (moneda==='usd' && esManual)` where `esManual = (metodo==='transferencia')`:
  - **USD + transferencia** → paid via PayPal/AstroPay in ACTUAL dollars, manual verification (`estado pendiente_verificacion`, `metodo 'transferencia'`, comprobante required). Stores `precio_cop = 0`, `precio_texto = '$X USD'`, NO Wompi signature. This is the only path where USD is the charged amount.
  - **USD + Wompi** → charges the NATIVE COP price exactly like a COP payment (commission gross-up + integrity signature, `estado pendiente_pago`). The UI must NOT show the conversion — let Wompi's checkout display the COP amount (`refrescarTotal` shows a "el total se mostrará en Wompi" notice, not a COP figure).
  - **COP** → unchanged.
  **Why:** the user wanted USD customers to still be able to use Wompi without us computing or showing any conversion rate (an earlier TRM-based conversion confused/overcharged them). **How to apply:** mirror the `esManual`/`usdManual` split in BOTH `server.js` and `assets/wompi.js esManual()`; the USD price still recomputes in parallel with COP for discount/adelanto (`$X COP · Y USD` in `precio_texto`, slice cap ~180). `precio_cop=0` for USD-manual means those sales are excluded from the admin COP revenue sum (accepted tradeoff — mixed currencies can't sum).

- **USD amounts in the manual path are validated against the catalog like COP is.** `USD_VALIDOS` is a Set built at startup by scraping every `N USD` value from `index.html` (mirrors `PRECIOS_VALIDOS` for COP). In `/api/booking`, when `usdManual`, the BASE usd (captured before discount/adelanto, so it's a single catalog value) must be in `USD_VALIDOS` or the request is rejected. **Why:** the manual USD amount is client-supplied; without this a customer could underpay by sending an arbitrary dollar figure. **How to apply:** validate the base (pre-modifier) USD, not the discounted/adelanto total; catalog USD prices include small values like `$1 USD` and add-ons like `$4 USD` (velación) — all legitimate.

- **Wompi commission + USD pricing: `precio_cop` now stores the FINAL CHARGED gross, not the service price.** For Wompi payments the customer is charged a grossed-up amount so the merchant nets the service price: `X = ceil((base + 700*1.19) / (1 - 0.0265*1.19))` (= 2.65% + $700 + 19% IVA on the commission). Commission applies ONLY to Wompi, never transferencia. When USD is paid via Wompi the commission is applied to the service's NATIVE COP price (no conversion); USD via transferencia has no commission (`precio_cop=0`, see USD entry above). **Why:** the signature and `/gracias` verification both key off `precio_cop*100`, so the signed amount, stored `precio_cop`, and Wompi `amount-in-cents` must all equal this final charged total — storing the service price instead would break verification. The service-price context is preserved only in `precio_texto` (breakdown appended). **How to apply:** any change to the charge math must update `calcularCobro`/`conComisionWompi` in `server.js` AND mirror it in `assets/wompi.js` (server stays authoritative via `res.data.precio_cop`); admin revenue sum uses `precio_cop` so it now reflects gross charged, not net.

- **A new booking field must be threaded through SIX places or it silently diverges:** frontend modal (`assets/wompi.js`: input + FormData append + validation), `/api/booking` (`server.js`: parse/validate/store), and four views/outputs in the two lib files — `paginaGracias` (customer confirmation page), `correoNotificacion` (email), `adminDashboard` (table header + `<td>`) in `lib/templates.js`, plus BOTH `generarExcel` and `generarPDF` in `lib/exportar.js` (each is a separate hand-built layout with its own column count + width array). **Why:** review repeatedly caught the PDF and the confirmation page being missed. `lib/templates.js` has 3 distinct views that all render booking data — don't confuse the email block with the confirmation page.
