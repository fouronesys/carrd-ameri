# Fresatanika (Carrd-based catalog â†’ Node/Express)

- **Data layer is now PostgreSQL (`lib/db.js` = pg Pool, JSONB `data` column per table), NOT JSON files.** All `db.*` are async and MUST be awaited (every route handler + inner multer callback is async). Sessions are Postgres-backed via `connect-pg-simple` (`session` table, `createTableIfMissing`). The JSON files under `DATA_DIR` are kept only as a one-time migration source + backup. **Why:** enables zero-downtime multi-instance CapRover rolling updates (JSON-file locking couldn't). **How to apply:** never reintroduce a sync JSON read/write for live data; add new tables in `crearEsquema` with a JSONB `data` column.

- **JSONâ†’PG migration is marker-based idempotent (`meta` table, one `migracion_<tabla>` row per table).** `migrarDesdeJSON({soloSiVacio})`: on startup (`soloSiVacio=true`) it imports a table only if unmarked, then marks it; the manual script `npm run migrar` (`scripts/migrar-a-postgres.js`) forces `soloSiVacio=false` to retry all. ALL inserts use `ON CONFLICT DO NOTHING` (accesos has a UNIQUE index on `(data->>'id')`; id-less legacy accesos get a synthesized stable `legacy-<index>` id so none are dropped), so re-runs never duplicate and a deleted row is never resurrected. **Why:** count-based skip risked partial-import data loss + accesos duplication on re-run. **How to apply:** any new migrated table needs its own `ON CONFLICT` target + `marcarMigracion` call, and a case in `verificarMigracion`.

- **`db.verificarMigracion()` proves no data loss; the manual script fails non-zero on mismatch.** It counts valid source-JSON records per entity vs destination rows and passes only when `destino >= origen` (NOT strict equality â€” live traffic legitimately grows the destination after migration, so equality would false-alarm). `npm run migrar` runs it and throws if any table lost rows. **How to apply:** never assert `destino === origen` for a live-writable table; the data-loss guarantee is one-directional (`>=`).

- **`SESSION_SECRET` is REQUIRED in production (`obtenerSecretoSesion` throws at startup if unset when `NODE_ENV==='production'`); the random per-file fallback (`DATA_DIR/.session_secret`) is dev-only.** **Why:** multiple instances behind the balancer must sign the session cookie with the SAME secret, and a per-instance/random secret would invalidate sessions on restart or across instances. **How to apply:** set `SESSION_SECRET` as a shared env secret in CapRover; never rely on the file fallback in prod.

- **`/healthz` (checks `db.verificarConexion`) + graceful shutdown enable rolling updates.** SIGTERM/SIGINT â†’ `apagar()`: clearInterval, `server.close`, await in-flight background jobs (`tareaLimpieza`/`tareaSeguimientos` â€” launched via `lanzarLimpieza`/`lanzarSeguimientos` so the latest promise is always awaitable), then `db.cerrar()`, with a 10s force-exit timeout. Startup is wrapped in async `iniciar()` (awaits `db.init()` then `app.listen`). Dockerfile has a `HEALTHCHECK` hitting `/healthz`. **How to apply:** any new setInterval background job must store its promise and be awaited in `apagar()` before the pool closes; CapRover start-first order + parallelism are set in the dashboard Service Update Override (not in repo).

- **better-sqlite3 does NOT compile in this Replit environment** (native build fails). Postgres (pg, pure JS) is used instead. **Why:** repeated native build failures wasted time.

- **Uploaded photos still live under `DATA_DIR`/uploads** (only the structured data moved to Postgres; sessions moved to PG too). In dev `DATA_DIR` defaults to a local `data/` dir; in the Dockerfile it is `/data`.

- **CapRover deploy requires a persistent volume mounted at `/data` (for uploaded photos) AND a PostgreSQL database (`DATABASE_URL`).** **Why:** without the volume, redeploys wipe uploaded photos; without Postgres the app won't start. Structured data + sessions are now in Postgres, so those survive redeploys independently of the volume.

- **Secrets are required, not defaulted.** `ADMIN_PASSWORD` (admin panel at `/admin`) and `SMTP_PASS` (Zoho `admin@fourone.com.do`) must be set as env/secrets. The admin password fallback was intentionally removed so a missing env fails safe instead of exposing a predictable password.

- **Wompi Web Checkout requires an integrity signature or it rejects the tx** with "Firma de integridad requerida no enviada". Append `&signature:integrity=<hash>` to the checkout URL, where `hash = SHA256(reference + amountInCents + currency + WOMPI_INTEGRITY_SECRET)` (hex). Compute it SERVER-SIDE in `/api/booking` (needs the secret) and return it in the JSON; the frontend only appends it. **Why:** the signed `amountInCents`/`reference`/`currency` MUST exactly equal what the redirect URL sends (both `precio*100` / `ref` / `'COP'`) or Wompi rejects it. The `test_integrity_*` secret pairs with `pub_test_*`, `prod_integrity_*` with `pub_prod_*` â€” swap BOTH together when going live. Only sign Wompi payments, not transferencia. Fail-closed: if `metodo=wompi` but the secret is unset, return 503 instead of redirecting to a checkout that will error.

- **Payment integrity:** `/api/booking` accepts a price only if it appears in the catalog (prices are parsed from `index.html` at startup). `/gracias/:ref` marks a booking as `agendado` ONLY when the Wompi tx is APPROVED **and** `reference`, `amount_in_cents` (== precio_cop*100), and `currency === 'COP'` all match exactly. Never loosen these checks.

- **Photo uploads are raster-only** (jpeg/png/webp; SVG blocked) and served with `X-Content-Type-Options: nosniff` + `Content-Security-Policy: default-src 'none'; sandbox` to prevent stored-XSS in the authenticated admin view.

- **CSRF on admin POST routes uses Origin/Referer validation** (a `mismoOrigen` middleware), not a token library. **Why:** the whole UI is plain HTML built via string concatenation in `lib/templates.js`, so per-form hidden-token plumbing is heavy; combined with the `SameSite=Lax` session cookie, Origin/Referer matching is sufficient defense for the destructive actions. **How to apply:** any new state-changing admin POST must be guarded by `mismoOrigen` (and `requiereAdmin`).

- **Any new uploaded-file field must also be threaded through the delete/cleanup path** in `lib/db.js` (`eliminarAdjuntos`, used by both `eliminarPorRef` and `eliminarPendientesAntiguos`), or the file becomes an orphan on disk. **Why:** `comprobante` was added everywhere except deletion, leaving sensitive payment-proof files behind after admin deletes a booking. **How to apply:** every field that stores an uploaded filename must be unlinked when its record is removed.

- **Scheduled reminders run in-process via `setInterval` in the `app.listen` callback** (like `limpiarPendientes`), guarded by an in-process boolean flag to avoid overlap on slow SMTP. Follow-up emails (5 semanas / 4 meses after `trabajo_hecho_en`) go to the ADMIN inbox (`NOTIFY_TO`/`SMTP_USER`), not the client â€” contacto is usually a social handle, not an email. **Why:** no external cron/queue in this CapRover setup. **How to apply:** persist a per-record "sent" timestamp (e.g. `seguimiento_5s_en`) set only after `r.enviado`, so a resend never fires; these internal tracking flags are the EXCEPTION to the six-place threading rule below â€” they are NOT rendered in any view/export.

- **Discounts (promotions/codes) apply to HECHIZOS ONLY, and the server must verify the booking really is that hechizo â€” not just trust the client-sent `hechizo_clave`.** The catalog is parsed from `#hechizos-section` of `index.html` at startup into a claveâ†’hechizo map. Duplicate hechizo names with different prices exist (e.g. "Corte de Lazos"), so each map entry keeps a `precios` array of ALL its catalog prices; a discount is applied only if the client-sent base price is a member of that hechizo's `precios`. **Why:** without the price-membership check, a client could book a non-hechizo service at its own valid price but send a valid `hechizo_clave` and steal a hechizo-only discount. **How to apply:** never gate a discount on the clave alone; the (clave, base-price) pair must both match the catalog.

- **Discounts never stack â€” take the larger of date-promo % vs promo-code %** (`Math.max`). The final price is always recomputed server-side in `/api/booking`; the frontend only sends the ORIGINAL base price + clave + code and then uses the server's returned `precio_cop` for the Wompi `amount-in-cents`. Promo codes are admin-only to create/delete (`soloAdmin`); date promos and everything else are allowed for assistants.

- **Promo codes support an optional usage limit (`limite_usos`, 0 = ilimitado).** Usage is counted DYNAMICALLY, not with a stored counter: `db.contarUsosCodigo(codigo)` counts agendamientos whose `codigo_promo` matches (case-insensitive) and whose `estado !== 'rechazado'`. `codigoDescuentoPara` returns `{valido:false, agotado:true}` when `usos >= limite`; the admin dashboard attaches `usos` per code for display (`X/N usos` Â· agotado/ilimitado). **Why dynamic:** self-healing â€” rejected payments and `eliminarPendientesAntiguos` cleanup free slots automatically, no counter to keep in sync across the six booking paths. **Accepted tradeoff:** the check+create is NOT atomic (separate JSON files, no lock), so simultaneous checkouts can oversubscribe by 1 â€” acceptable for this low-traffic single shop, same best-effort nature as every other write here; do NOT add a mutex/lock layer unless traffic demands it.

- **Two admin roles (`admin` / `asistente`), stored as `session.rol`, gated by `soloAdmin` middleware.** Login now supports NAMED USERS: admins create accounts (`usuario`+`nombre`+`rol`+password) via `/admin/usuarios`, verified with scrypt+timingSafeEqual (`verificarPassword`); the legacy env-password path (`ADMIN_PASSWORD`â†’admin) is a fallback when no user matches. Admin-only actions: creating/deleting promo codes, deleting agendamientos marked `trabajo_hecho`, order editing (`/admin/editar`, `/admin/servicio`), evidence upload (`/admin/evidencia`), user management (`/admin/usuarios`), AND **all payment-status mutation** â€” `/admin/aprobar` + `/admin/rechazar` are `soloAdmin` and their approve/reject buttons are hidden from assistants (replaced by a "đź”’ Solo admin" note). **Why:** the admin-panel-upgrade task made payment-status editing admin-only, superseding the earlier behavior where assistants could approve/reject. **How to apply:** any route that changes payment/estado must be `soloAdmin` AND mirror the restriction in `adminDashboard` via `esAsistente` so the UI matches the backend.

- **Login audit log + email alert on EVERY login attempt (success AND failure).** `db.registrarAcceso` stores {usuario, rol, exito, ip, agente, cuando} capped at 500 (unshift); `registrarYalertarAcceso(req,{...})` in server.js also fires `mailer.enviarAlertaLogin` fire-and-forget (`.catch` swallow). Rendered in `adminDashboard` "Historial de accesos" (admin-only). **How to apply:** call `registrarYalertarAcceso` from every auth branch (named-user success, env-password success, and failure).

- **USD is a NATIVE catalog price, never a conversion. There is NO TRM/exchange-rate logic anywhere (fully removed).** Wompi charges COP only (confirmed official docs). Each service has independent COP and USD prices parsed from `index.html`. The charge depends on the currency+method combo, keyed in `server.js /api/booking` off `usdManual = (moneda==='usd' && esManual)` where `esManual = (metodo==='transferencia')`:
  - **USD + transferencia** â†’ paid via PayPal/AstroPay in ACTUAL dollars, manual verification (`estado pendiente_verificacion`, `metodo 'transferencia'`, comprobante required). Stores `precio_cop = 0`, `precio_texto = '$X USD'`, NO Wompi signature. This is the only path where USD is the charged amount.
  - **USD + Wompi** â†’ charges the NATIVE COP price exactly like a COP payment (commission gross-up + integrity signature, `estado pendiente_pago`). The UI must NOT show the conversion â€” let Wompi's checkout display the COP amount (`refrescarTotal` shows a "el total se mostrarĂˇ en Wompi" notice, not a COP figure).
  - **COP** â†’ unchanged.
  **Why:** the user wanted USD customers to still be able to use Wompi without us computing or showing any conversion rate (an earlier TRM-based conversion confused/overcharged them). **How to apply:** mirror the `esManual`/`usdManual` split in BOTH `server.js` and `assets/wompi.js esManual()`; the USD price still recomputes in parallel with COP for discount/adelanto (`$X COP Â· Y USD` in `precio_texto`, slice cap ~180). `precio_cop=0` for USD-manual means those sales are excluded from the admin COP revenue sum (accepted tradeoff â€” mixed currencies can't sum).

- **USD amounts in the manual path are validated against the catalog like COP is.** `USD_VALIDOS` is a Set built at startup by scraping every `N USD` value from `index.html` (mirrors `PRECIOS_VALIDOS` for COP). In `/api/booking`, when `usdManual`, the BASE usd (captured before discount/adelanto, so it's a single catalog value) must be in `USD_VALIDOS` or the request is rejected. **Why:** the manual USD amount is client-supplied; without this a customer could underpay by sending an arbitrary dollar figure. **How to apply:** validate the base (pre-modifier) USD, not the discounted/adelanto total; catalog USD prices include small values like `$1 USD` and add-ons like `$4 USD` (velaciĂłn) â€” all legitimate.

- **Wompi commission + USD pricing: `precio_cop` now stores the FINAL CHARGED gross, not the service price.** For Wompi payments the customer is charged a grossed-up amount so the merchant nets the service price: `X = ceil((base + 700*1.19) / (1 - 0.0265*1.19))` (= 2.65% + $700 + 19% IVA on the commission). Commission applies ONLY to Wompi, never transferencia. When USD is paid via Wompi the commission is applied to the service's NATIVE COP price (no conversion); USD via transferencia has no commission (`precio_cop=0`, see USD entry above). **Why:** the signature and `/gracias` verification both key off `precio_cop*100`, so the signed amount, stored `precio_cop`, and Wompi `amount-in-cents` must all equal this final charged total â€” storing the service price instead would break verification. The service-price context is preserved only in `precio_texto` (breakdown appended). **How to apply:** any change to the charge math must update `calcularCobro`/`conComisionWompi` in `server.js` AND mirror it in `assets/wompi.js` (server stays authoritative via `res.data.precio_cop`); admin revenue sum uses `precio_cop` so it now reflects gross charged, not net.

- **Cart / combined-order flow (`/api/checkout`, `/pedido/:ref`) is COP-only and creates one booking per service sharing a `pedido_ref`.** Client accounts are optional (email+password hashed with `crypto.scryptSync` + `timingSafeEqual`, pure Node core â€” NOT bcrypt, which won't compile here); `req.session.cliente` is fully independent of `req.session.admin` (logout deletes only its own key). Cart lives in browser `localStorage`; on login the browser cart merges with the server-saved cart (dedup by firma `nombre|precio_cop|hechizo_clave|es_extra|es_adelanto`). **Why:** USD-in-cart and per-item promo codes/adelanto-add were deliberately left OUT of the cart (only auto date-promo applies) to keep the single-payment math tractable â€” the single-service flow still supports all of those. **How to apply:** per-booking `precio_cop` = NET service price (post date-discount, NO commission); the shared `pedido_total_cop` = gross order total (sum of nets + Wompi commission applied ONCE over the whole order). Wompi signature/`/pedido` verification key off `pedido_total_cop*100` with `reference === pedido_ref`. Admin approve/reject and the paid-mark cascade to ALL siblings via `db.actualizarPorPedido`; `clientes.json` is `{clientes:[...]}` (not a bare array).

- **Two distinct email audiences: ADMIN vs CLIENT.** All the original notifications (`enviarNotificacion`, `enviarNotificacionPedido`, follow-ups) go to the ADMIN inbox and fire for every booking. Client-facing mail (`enviarEstadoCliente` status alerts, `enviarEvidenciaCliente`) is NET-NEW and only possible for REGISTERED clients â€” resolve the address via `emailDelCliente(reg)` (= `db.obtenerClientePorId(reg.cliente_id).email`); guests have no email on file (`contacto` is a WhatsApp/social handle, not email) so they get a "crea tu cuenta" recommendation on the thank-you pages instead. **Why:** bookings only carry `cliente_id` when made while logged in; sending to `contacto` would bounce. **How to apply:** never send client mail without a resolved account email; keep admin mail flow untouched when adding client mail.

- **Client status alerts are deduped with per-state timestamp flags** `cliente_aviso_agendado` / `_rechazado` / `_realizado`, set only after `r.enviado`. Same pattern as the follow-up `seguimiento_*_en` flags: internal tracking fields, NOT rendered in any view/export (exception to the six-place threading rule). Wired into `/admin/aprobar` `/rechazar` `/trabajo` AND both Wompi auto-agendado blocks (`/gracias/:ref`, `/pedido/:pedidoRef`); pedido variants use `notificarEstadoPedido` (one mail for the whole order, flag on all siblings). **How to apply:** any new automatic client alert needs its own dedup flag and must hook every path that can trigger that state (manual admin routes + Wompi callback).

- **Work-evidence (`evidencia_fotos[]` + `evidencia_notas`) has a public token-gated share link.** Admin uploads via `POST /admin/evidencia/:ref` (reuses the shared multer `upload`, jpg/png/webp, 8MB, `upload.array('fotos',10)`); a random `evidencia_token` (16 bytes hex) is minted once and grants public read at `GET /evidencia/:ref/:token` (page) and `/evidencia-foto/:ref/:token/:nombre` (image, with `path.basename` + membership check in `evidencia_fotos` + nosniff/CSP headers). This lets the admin share evidence by WhatsApp even for GUEST bookings (no account); registered clients also get it emailed with the images as `cid:evidencia0..N` inline attachments. `evidencia_fotos` IS threaded through `eliminarAdjuntos`. **How to apply:** treat the token as the only auth for the public routes â€” never expose evidence via `/admin/foto` semantics to non-admins, and keep the filename-membership check so an arbitrary upload name can't be fetched with a valid token.

- **A new booking field must be threaded through SIX places or it silently diverges:** frontend modal (`assets/wompi.js`: input + FormData append + validation), `/api/booking` (`server.js`: parse/validate/store), and four views/outputs in the two lib files â€” `paginaGracias` (customer confirmation page), `correoNotificacion` (email), `adminDashboard` (table header + `<td>`) in `lib/templates.js`, plus BOTH `generarExcel` and `generarPDF` in `lib/exportar.js` (each is a separate hand-built layout with its own column count + width array). **Why:** review repeatedly caught the PDF and the confirmation page being missed. `lib/templates.js` has 3 distinct views that all render booking data â€” don't confuse the email block with the confirmation page.
